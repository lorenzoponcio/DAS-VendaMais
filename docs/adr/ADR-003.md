# ADR-003: Seleção de Biblioteca para Acesso ao Banco de Dados (pymssql vs SQLAlchemy)

**Status:** Aceito  |  **Data:** 2026-06-09  |  **Autores:** Carlos Augusto Brazão da Silva, Juliana Seemann, Lorenzo Poncio, Matheus Henrique Faria Motta

## Contexto

Foi realizada uma Prova de Conceito (PoC) para comparar empiricamente o desempenho de duas abordagens de acesso ao banco de dados SQL Server em uma Azure Function:

- **Driver nativo:** `pymssql`
- **ORM/Abstração:** `SQLAlchemy`

A comparação utilizou o mesmo banco de dados, a mesma tabela (`erp.pedido_item`) e a mesma consulta SQL:

```sql
SELECT TOP 5 * FROM erp.pedido_item
```

O objetivo foi medir o tempo necessário para consultar e trazer os registros para memória, considerando o tempo de execução da query e o tempo total da operação (criação de conexão + execução + retorno dos dados).

## Decisão

A biblioteca `pymssql` foi escolhida para operações de acesso ao banco de dados no contexto atual.

Os resultados da PoC demonstraram que o tempo de execução da query foi semelhante entre as duas bibliotecas (diferença média de ~1,6 ms). Entretanto, no tempo total de operação, o `pymssql` apresentou média de **0,3321 s** frente a **0,8067 s** do `SQLAlchemy` — aproximadamente **2,4 vezes mais rápido**. Esse overhead do SQLAlchemy é esperado e decorre da sua camada adicional de abstração (configuração de engine, mapeamento de sessão e gerenciamento de metadados), que não agrega valor para consultas SQL diretas e simples.

### Resultados da PoC

#### pymssql

| Execução   | Tempo de Query | Tempo Total  |
|------------|----------------|--------------|
| Execução 1 | 0,0176 s       | 0,3490 s     |
| Execução 2 | 0,0179 s       | 0,3151 s     |
| **Média**  | **0,0178 s**   | **0,3321 s** |

#### SQLAlchemy

| Execução   | Tempo de Query | Tempo Total  |
|------------|----------------|--------------|
| Execução 1 | 0,0232 s       | 0,8334 s     |
| Execução 2 | 0,0156 s       | 0,7800 s     |
| **Média**  | **0,0194 s**   | **0,8067 s** |

#### Comparativo consolidado

| Biblioteca | Tipo          | Média Query  | Média Total  |
|------------|---------------|--------------|--------------|
| pymssql    | Driver Nativo | 0,0178 s     | 0,3321 s     |
| SQLAlchemy | ORM           | 0,0194 s     | 0,8067 s     |

## Consequências

### (+) Benefícios

- (+) Menor tempo total de execução (~2,4× mais rápido que o SQLAlchemy);
- (+) Menor overhead de inicialização, relevante no modelo serverless onde conexões são recriadas a cada execução;
- (+) Implementação simples e direta para consultas SQL sem necessidade de mapeamento de objetos;
- (+) Menor número de dependências no projeto;

### (-) Pontos negativos

- (-) Menor abstração, exigindo SQL manual em todas as operações;
- (-) Menor flexibilidade para modelagem de dados complexa ou reutilização de lógica de acesso;
- (-) Maior acoplamento direto às consultas SQL, dificultando eventuais migrações de banco de dados;

## Alternativas rejeitadas

- **SQLAlchemy (ORM):**
  Rejeitado para o contexto atual devido ao overhead significativo no tempo total de operação (~0,8067 s de média), aproximadamente 2,4 vezes superior ao `pymssql`. Continua sendo uma alternativa válida para módulos futuros que demandem maior complexidade de modelagem, múltiplos bancos de dados, ou onde manutenibilidade e reutilização de código sejam priorizadas em detrimento do desempenho bruto.
