# ADR-001: Estratégia de Ingestão de Dados (Serverless com Azure Functions)

**Status:** [Aceito]  |  **Data:** [2026-04-06]  |  **Autores:** [Carlos Augusto Brazão da Silva,Juliana Seemann, Lorenzo Poncio, Matheus Henrique Faria Motta]
## Contexto

A VendaMais Distribuidora enfrenta dificuldades na obtenção de indicadores operacionais atualizados. Atualmente, os relatórios são gerados manualmente, com um gap que pode chegar a até 30 dias.
O sistema ERP vigente centraliza dados transacionais como vendas, estoque, financeiro e logística, porém não disponibiliza uma camada eficiente para extração, processamento e análise desses dados.
Diante disso, propõe-se a criação de um pipeline de dados estruturado nas etapas de **ingestão → armazenamento → transformação → consumo**, com o objetivo de reduzir a defasagem das informações para no máximo 24 horas.

## Decisão

A escolha por Azure Functions no modelo serverless foi feita devido ao baixo custo operacional, facilidade de integração com serviços Azure e capacidade de execução automatizada via agendamento.
Comparado às alternativas, o modelo serverless elimina a necessidade de gerenciamento de infraestrutura (como em máquinas virtuais e containers), reduzindo a complexidade operacional.
A função será executada de forma agendada (**timer trigger**), conectando-se ao ERP, extraindo os dados do dia e armazenando-os no **Azure Blob Storage**.

## Consequências

### (+) Benefícios:
- (+) Baixo custo operacional, pois o modelo serverless cobra apenas por execução  ;
- (+) Escalabilidade automática, suportando variações no volume de dados sem necessidade de configuração manual  ;
- (+) Simplicidade de implementação e integração nativa com o ecossistema Azure  ;
- (+) Redução de esforço operacional (sem necessidade de gerenciar servidores)  ;
- (+) Alta aderência ao modelo de pipeline de dados proposto  ;

### (-) Pontos negativos

- (-) Menor controle sobre infraestrutura comparado a VMs ou containers  ;
- (-) Possíveis limitações de tempo de execução (timeout) em cargas maiores  ;
- (-) Dependência do ecossistema Azure (vendor lock-in);

## Alternativas rejeitadas

- **Máquinas Virtuais (VMs na Azure):**  
  Rejeitada devido ao alto custo e necessidade de gerenciamento manual (infraestrutura, escalabilidade, manutenção).

- **Containers (Azure Container Instances ou Kubernetes):**  
  Rejeitada por maior complexidade operacional e overhead desnecessário para um processo simples e agendado.

- **Execução manual / scripts locais:**  
  Rejeitada por não atender ao requisito de automação, além de ser propensa a erros humanos e inconsistência.
